# Задание 2.1: Тесты API

## [BUG-01] Несоответствие схемы JSON-ответа спецификации
- **ID:** BUG-01
- **Приоритет (Priority):** High (P1)
- **Серьезность (Severity):** Major
- **Компонент:** API / Create Item
- **Описание:** Структура тела ответа при успешном создании объявления (POST) не соответствует документации Postman-коллекции.
- **Шаги воспроизведения:**
  1. Отправить POST запрос на `/api/1/item` с валидным JSON-телом.
- **Фактический результат:** Приходит объект вида `{"status": "Сохранили объявление - {uuid}"}`.
- **Ожидаемый результат:** Приходит полный объект сущности с полями `id`, `sellerId`, `name`, `price`, `statistics`, `createdAt`.

## [BUG-02] Отсутствие валидации отрицательных значений в поле 'price'
- **ID:** BUG-02
- **Приоритет (Priority):** High (P1)
- **Серьезность (Severity):** Major
- **Компонент:** API / Validation
- **Описание:** Система позволяет создать объявление, цена которого меньше нуля, что нарушает бизнес-логику.
- **Шаги воспроизведения:**
  1. Отправить POST запрос на `/api/1/item`.
  2. В теле запроса указать `"price": -100`.
- **Фактический результат:** Статус `200 OK`, объявление успешно создано.
- **Ожидаемый результат:** Статус `400 Bad Request` с информативным сообщением об ошибке валидации.

---

## [BUG-03] Нарушение идемпотентности метода POST (Дублирование ID)
- **ID:** BUG-03
- **Приоритет (Priority):** Medium (P2)
- **Серьезность (Severity):** Medium
- **Компонент:** API / Business Logic
- **Описание:** При повторной отправке идентичных данных метод POST не создает новую сущность, а возвращает ID предыдущей.
- **Шаги воспроизведения:**
  1. Отправить POST на `/api/1/item` с данными (A). Сохранить ID_1.
  2. Повторно отправить тот же запрос с данными (A). Сохранить ID_2.
- **Фактический результат:** ID_1 равен ID_2.
- **Ожидаемый результат:** Создание новой сущности (ID_1 != ID_2) ИЛИ возврат ошибки о конфликте (409 Conflict).

## [BUG-04] Слабая типизация поля 'price' (прием строк)
- **ID:** BUG-04
- **Приоритет (Priority):** Medium (P2)
- **Серьезность (Severity):** Major
- **Компонент:** API / Type Validation
- **Описание:** Эндпоинт POST `/api/1/item` принимает строковое значение (String) в поле `price`, в то время как ожидается целое число (Integer).
- **Шаги воспроизведения:**
  1. Отправить POST на `/api/1/item`.
  2. В теле запроса указать `"price": "1000"`.
- **Фактический результат:** Статус `200 OK`, ресурс успешно создан.
- **Ожидаемый результат:** Статус `400 Bad Request`. Типы данных должны строго соответствовать спецификации (Integer).

---

## [BUG-05] Отсутствие валидации отрицательных значений в объекте 'statistics'
- **ID:** BUG-05
- **Приоритет (Priority):** High (P1)
- **Серьезность (Severity):** Major
- **Компонент:** API / Validation
- **Описание:** Сервис принимает отрицательные значения для счетчиков лайков, просмотров и контактов.
- **Шаги воспроизведения:**
  1. Отправить POST на `/api/1/item`.
  2. В объекте `statistics` указать `"likes": -5`.
- **Фактический результат:** Статус `200 OK`.
- **Ожидаемый результат:** Статус `400 Bad Request`. Значения статистики не могут быть меньше нуля.

---

## [BUG-06] Ошибка валидации граничного значения (likes: 0)
- **ID:** BUG-06
- **Приоритет (Priority):** Blocker (P0)
- **Серьезность (Severity):** Critical
- **Компонент:** API / Validation
- **Описание:** Система воспринимает число `0` как отсутствие значения в обязательном поле.
- **Шаги воспроизведения:**
  1. Отправить POST на `/api/1/item`.
  2. В объекте `statistics` указать `"likes": 0`.
- **Фактический результат:** Статус `400 Bad Request`, сообщение: "поле likes обязательно".
- **Ожидаемый результат:** Статус `200 OK`. Новое объявление создано с нулевым количеством лайков.

---

## [BUG-07] Отсутствие валидации нулевого значения в поле 'price'
- **ID:** BUG-07
- **Приоритет (Priority):** Medium (P2)
- **Серьезность (Severity):** Normal
- **Компонент:** API / Validation
- **Описание:** Система позволяет создать объявление с ценой, равной 0. Это нарушает бизнес-логику (цена товара должна быть положительным числом).
- **Шаги воспроизведения:**
  1. Отправить POST запрос на `/api/1/item` с валидным телом.
  2. В поле `price` указать значение `0`.
- **Фактический результат:** Статус `400 Bad Request`, объявление не создано.
- **Ожидаемый результат:** Статус `200 OK`. Можно создать объявление с бесплатной услугой.

---

## [BUG-08] Метод DELETE недоступен (Method Not Allowed)
- **ID:** BUG-08
- **Приоритет (Priority):** Medium (P2)
- **Серьезность (Severity):** Major
- **Компонент:** API / HTTP Methods
- **Описание:** Согласно документации (Postman-коллекция), ресурс должен поддерживать удаление через DELETE, но сервер блокирует вызов.
- **Шаги воспроизведения:**
  1. Создать объявление, получить его ID.
  2. Отправить DELETE запрос на `/api/1/item/{id}`.
- **Фактический результат:** Статус `405 Method Not Allowed`.
- **Ожидаемый результат:** Статус `200 OK` или `204 No Content`, объявление удалено.

---

## [BUG-09] Критическая ошибка удаления (Data Persistence)
- **ID:** BUG-09
- **Приоритет (Priority):** Blocker (P0)
- **Серьезность (Severity):** Critical
- **Компонент:** API / Persistence
- **Описание:** Удаленный ресурс остается доступным для чтения (GET) после успешного выполнения DELETE.
- **Шаги воспроизведения:**
  1. Удалить объявление через `DELETE /api/1/item/{id}`. Получить статус 200.
  2. Отправить `GET /api/1/item/{id}` для этого же ID.
- **Фактический результат:** GET возвращает `200 OK` и полное тело "удаленного" объявления.
- **Ожидаемый результат:** GET должен возвращать `404 Not Found` или `410 Gone`.

## [BUG-10] Некорректный статус-код при удалении несуществующего ресурса
- **ID:** BUG-10
- **Приоритет (Priority):** Medium (P2)
- **Серьезность (Severity):** Minor
- **Компонент:** API / Error Handling
- **Описание:** При попытке удаления несуществующего ID или ID неверного формата сервер возвращает код 405 вместо кодов ошибки ресурса (404/400).
- **Шаги воспроизведения:**
  1. Отправить DELETE запрос на `/api/1/item/{invalid_id}` (например, несуществующий UUID).
- **Фактический результат:** Статус `405 Method Not Allowed`.
- **Ожидаемый результат:** Статус `404 Not Found` (если ресурс не найден) или `400 Bad Request` (если неверный формат ID).

- **ID:** BUG-11
- **Приоритет (Priority):** Low (P3)
- **Серьезность (Severity):** Minor
